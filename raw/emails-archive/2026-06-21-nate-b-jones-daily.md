## The Big Idea
Agent ownership is the skill most teams skip in 2026. The fastest way to make an agent dangerous isn't malicious code — it's letting everybody use it and nobody own it. The moment an agent starts reading real files, drafting real messages, and changing things other people rely on, it stops being a tool you use and becomes work you are responsible for. Nate's bet: the teams that name an owner for every agent pull ahead of the teams still collecting demos.

## The Framework
- **The one-sentence ownership test:** the rule for exactly when an agent crosses from convenient to consequential — and who has to own it. Not a committee, not "the AI team" by default. One person close enough to know if the agent is helping, drifting, or just adding polished noise.
- **Job, diet, boundaries, review loop:** the four fields every owned agent needs defined — what job it does, what it consumes, what it's not allowed to touch, and the cadence a human checks its work.
- **The Agent Owner's Card:** a one-page, human-readable artifact — the counterpart to the machine-readable cards agents already exchange with each other — that makes ownership visible. Paired with a prompt you point an existing agent at so it drafts the fields it can see and hands the ownership questions back to a human to answer and review.
- **Not IT governance:** a committee can govern the road; it can't replace one accountable person owning each agent actually doing the work.

## Where It Applies
- Any agent that crossed from demo to depended-on: support agents answering customer questions, triage/planning agents turning tickets into priorities, product-team agents doing recurring prep work.
- Scales differently at three levels: a personal agent (you're the obvious owner), a shared team agent (ownership diffuses fast), and a multi-agent pipeline (failures compound silently across handoffs) — each breaks in a different way.
- The "haunted house" risk: automated systems still running long after the reason for them existed, quietly producing drafts and recommendations nobody is checking.

## Receipts
- The stale-policy support agent: a support agent that keeps answering from a policy that changed last quarter — wrong, but not loudly wrong, so nobody notices.
- The clean-looking-wrong planning agent: a planning agent that keeps turning noisy tickets into priorities that look clean and decisive but are quietly mis-ranked.
- The "haunted house" metaphor: Nate's name for a company full of agents still running long after their original purpose disappeared — output keeps arriving, value drains out, everyone stays busy.
- Machine cards precedent: the Agent Owner's Card is explicitly positioned as the human-readable counterpart to the machine-readable "agent cards" agents already exchange with each other in multi-agent systems.

## What It Means For You
1. **Run the one-sentence test today.** For every agent your team touches, ask: has it started reading real files, drafting real messages, or changing things people rely on? If yes, it needs a named owner now, not eventually.
2. **Assign one person, not a committee.** Ownership defaults to "the AI team" only by accident — make it deliberate and individual.
3. **Fill out an Owner's Card per agent.** Capture job, diet, boundaries, and review loop for every agent on your roster — point an existing agent at the prompt to draft what it can see, then have the human owner fill in and review the rest.
4. **Schedule the review loop like maintenance, not a launch task.** Treat ownership as the same ongoing discipline as the agent-maintenance habits from two weeks back — stale agents are how haunted houses start.

## Mistake to Avoid
Treating agent ownership as an IT governance problem — standing up a committee to write rules for "the road" instead of naming the one person who owns each agent actually doing the work. A policy document doesn't notice a support agent quoting last quarter's pricing; an owner does.
