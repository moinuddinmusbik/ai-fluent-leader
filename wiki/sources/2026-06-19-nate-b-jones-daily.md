# Your AI Skills Are Trapped | Here's How to Own Them

[[nate-b-jones]] · 2026-06-19 · FRAMEWORK · 17:46

> The prompt copies over. The intention copies over. The skill does not.
> — Substack, "Your skills are leaving your hands"

## The Big Idea

Agent skills — the procedures, standards, and taste that used to live in a person's hands — are migrating into software, the same way memory did before them. Nate frames this as the second act of an ownership fight he started with "Open Brain": Open Brain argued your memory should be yours; Open Skills argues the same about the way you work. Today almost none of that portability exists — the prompt and the intention carry over when you switch AI tools, but the skill itself does not, so you rebuild it from memory every time, a teammate quietly improves a different copy and never tells you, and your best workflow ends up stranded in a chat history nobody can find. He launches **Open Skills**, a public library of installable skills and runbooks, as the fix.

## The Framework

- **Prompt, Memory, Skill:** Nate separates three things people collapse into one — only the skill survives a model or tool change, and only if it's structured as a portable artifact rather than a one-off instruction.
- **Procedural debt:** he names four specific places this debt shows up in daily agent work (chapter "Four places procedural debt shows up," 1:39) — the common thread is work you have to keep re-explaining every time you switch context.
- **Skill vs. clever prompt:** a real skill is explicitly distinguished from a prompt that merely happens to work inside one tool (chapter "Prompt vs skill: search, voice, and browser QA," 6:00).
- **Verification as the proof layer:** a skill isn't "done" until it produces a check a human can inspect (chapter "Verification: don't call it done without proof," 11:09).

The umbrella concept is [[open-skills]]; its precursor is [[open-brain]].

## How It Works

- **Skills live as concrete artifacts:** `SKILL.md` files, runbooks, scripts, MCP configs, permission boundaries, and agent workflows — not chat history.
- **Primitives + composition:** small reusable skills combine into larger documented procedures (chapter "Skills as primitives, runbooks as composition," 8:46).
- **The flywheel:** every completed agent session becomes raw material for a new reusable skill instead of a one-off transcript (chapter "The flywheel: turning sessions into reusable skills," 11:58).
- **Open Brain + Open Skills stack together:** portable memory plus portable procedure is the full ownership picture (chapter "Open Brain plus Open Skills together," 12:53).

## Where It Applies

- **Cross-tool by design:** built to travel across Claude Code, Codex, Cursor, and ChatGPT — not to live inside one vendor's project format.
- **Concrete proof case:** a real support-billing workflow rebuilt so it survives the jump across Codex, Claude Code, and Cursor instead of breaking on contact.
- **Today's rule files don't travel:** Cursor's and Claude Code's own project-rule conventions don't carry over to each other (chapter "When Cursor and Claude Code rules don't travel," 3:04).
- **Launch vehicle:** Open Skills, a public library of installable skills and runbooks, ships the same day as this episode (chapter "What Open Skills is, launching today," 2:51).

## Receipts

- **Open Skills library:** launched June 19, 2026, alongside this episode — a public, installable starting set rather than a closed vendor feature.
- **Named proof-of-concept:** the support-billing workflow rebuild that now runs identically across Codex, Claude Code, and Cursor.
- **Named precursor:** "Open Brain," his earlier Substack piece making the parallel claim that your memory should be yours, not trapped inside whichever company's product captured it first.
- **Compounding cost:** he names it directly — every tool switch becomes a rebuild, every new hire starts from scratch, every improvement risks dying as one person's private habit.

## What It Means For You

1. **Inventory your team's agent procedures by tool.** For every workflow someone has gotten working in Claude Code, Codex, or Cursor, ask whether it would survive that person switching tools tomorrow — or whether it's stranded in one app.
2. **Convert your best workflows into portable artifacts, not prompts.** Write your highest-value procedures as `SKILL.md` files and runbooks rather than one-off chat instructions, so they move with you across tools and survive a model upgrade.
3. **Require a verification step before any skill counts as "done."** Build a checkable proof trail into every agent procedure — don't accept output that only asserts a conclusion.
4. **Treat career capital as something to own, not rent.** If skills built with AI are now part of how your team does its job, audit who actually controls that procedural knowledge — your people, or whichever vendor's tool it happened to get built in.

## Mistake to Avoid

Assuming that because a prompt and its intention carry over cleanly when you switch AI tools, the underlying skill did too. It doesn't — and that gap is what forces you to rebuild the same workflow from memory on every tool switch, lets a teammate's improvement die in a private copy nobody else sees, and leaves your best procedures stranded in a chat history no one can find again.
